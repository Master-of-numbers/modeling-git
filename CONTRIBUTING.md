# Виконання завдань

> Цей файл містить опис та рекомендації щодо оформлення звітів виконання завдань для здобувачів освіти.

## Опис

Завдання виконуються у вигляді нотаток Marimo, збережених у теці [code](./code) На кожне завдання - окремий файл,
названий за наступним принципом: `<номер завдання>_<номер за списком>.py`

Наприклад: `1_15.py`

Зразок структури файлу нотатника наведено в [example.py](./code/example.py).

## Розгортання середовища розробки

Розробка звітів здійснюється у браузерному середовищі Marimo.

Для розгортання Marimo рекомендується використовувати `uv`.

### Встановлення `uv`:

#### Windows

В `PowerShell` та виконайте наступну команду:

```shell
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"
```

#### Linux

```shell
curl -LsSf https://astral.sh/uv/install.sh | sh
```

### Встановлення Marimo:

```shell
uv tool install marimo
```

## Запуск Marimo

В теці [code](./code) виконуємо наступну команду:

```shell
marimo edit example.py --sandbox
```

## Рекомендації щодо оформлення звітів виконання:

### Відтворюваність

Ваш документ повинен читатися як зв'язна історія (обчислювальний наратив) і виконуватися послідовно.

Слідкуйте за тим, щоб логіка оповіді йшла зверху вниз: від постановки задачі через ініціалізацію змінних до обчислень та
висновків.

### Єдине джерело істини (Single Source of Truth)

Не дублюйте ті самі дані в різних частинах документа. Усі вхідні параметри, константи та варіанти завдань повинні
оголошуватися лише один раз на початку коду:

- ❌ Погано: Писати у коді `y = 5 * x`, пам'ятаючи, що `5` — це маса з умови. Потім в іншому місці знову вручну множити
  на `5`.
- ✅ Правильно: Оголосити змінну `MASS = 5` у блоці конфігурації. Далі по всьому тексту і коду посилатися виключно на
  змінну `MASS`. Якщо у вас зміниться варіант і маса стане рівною `10`, ви зміните лише один рядок коду, а весь звіт
  оновиться автоматично.

### Розділяйте теорію та обчислення

Використовуйте кожен інструмент за його прямим призначенням. Текстові блоки (Markdown) існують для пояснення логіки, а
блоки коду (Python) — для математики, відмалювання графіків, діаграм тощо.

- **Для формул** використовуйте **LaTeX**: Загальний вигляд рівняння пишіть у Markdown за допомогою LaTeX. Наприклад: За
  законом
  Ома $I = \frac{U}{R}$.
- **Для обчислень** використовуйте **код**: Не рахуйте нічого на калькуляторі, щоб потім вписати відповідь у текст!
  Доручіть це Python (бібліотекам numpy, scipy, sympy). Код має взяти змінні $U$ та $R$, поділити їх і зберегти в нову
  змінну.

### Автоматизований вивід результатів

Звіт не повинен містити "захардкоджених" таблиць із результатами, вбитих вручну.

- ❌ Погано: Вручну малювати Markdown-таблицю і вписувати туди проміжні значення, дивлячись на вивід у консолі.
- ✅ Добре: Зібрати масиви даних у коді та вивести їх за допомогою вбудованих функцій відображення
  таблиць Marimo[^1] або сторонніх бібліотек Python. Якщо алгоритм обчислення зміниться, таблиця оновиться автоматично.

### Динамічна інтеграція значень у текст (f-strings)

Коли вам потрібно написати висновок, не вписуйте цифри руками. У Marimo (через функцію md) можна вбудовувати значення
змінних прямо в текст.

Приклад правильного висновку (через код):

```python
print(f"Отже, максимальна напруга становить {max_voltage:.2f} В, що відповідає вимогам.")
```

Це гарантує, що текст висновку ніколи не розсинхронізується з реальними розрахунками.

### Візуалізація замість смітника з тексту

Ніколи не виводьте на екран масиви з сотнями чисел (наприклад, print(array)). Це засмічує звіт і не несе жодної користі
для читача. Якщо даних багато — будуйте графік (через вбудовані інструменти marimo, matplotlib, plotly або altair).

Графіки також мають будуватися виключно на основі обчислених змінних.
